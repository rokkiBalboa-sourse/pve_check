# PVE DemoExam Checker 2026

Скрипты для автоматической проверки выполнения задач демонстрационного экзамена по компетенции "Сетевое и системное администрирование" (СиСА 09.02.06).

Скрипты проверяют конфигурацию виртуальных машин Proxmox VE (PVE) через QEMU Guest Agent.

---

## Быстрый запуск на сервере PVE

Для запуска проверки достаточно выполнить одну команду на вашем сервере PVE под пользователем **root**:

```bash
bash <(curl -sSL https://raw.githubusercontent.com/rokkiBalboa-sourse/pve_check/main/run.sh)
```

### Что делает этот загрузчик (`run.sh`):
1. **Выбор источника:** Предлагает выбрать между GitHub и зеркалом Yandex.Disk (удобно для корпоративных сетей с фаерволом). Если выбор не сделан в течение 30 секунд, загрузка пойдет автоматически с GitHub.
2. **GitHub с Git:** Если выбран GitHub и на сервере установлен `git`, клонирует репозиторий в `/root/pve_checks` (или обновляет через `git pull`, если репозиторий уже склонирован).
3. **GitHub без Git:** Если `git` не установлен, скачивает файлы по одному через `curl`, выводя красивый графический прогресс-бар от 0% до 100%.
4. **Yandex.Disk:** Скачивает предварительно упакованный ZIP-архив через Yandex Cloud API, распаковывает его с помощью встроенного в PVE Python 3 и удаляет архив.
5. **Запуск:** Делает скрипты исполняемыми и запускает интерактивное меню `menu.sh`.

---

## Как настроить зеркало на Yandex.Disk

Если в вашей сети заблокирован GitHub, настройте скачивание с Яндекс.Диска:

1. **Создайте архив:** Запакуйте файлы вашего репозитория (папки `v1-v4`, скрипты `menu.sh` и `run.sh`) в ZIP-архив (например, `pve_check.zip`). 
   * *Важно:* Файлы должны лежать в корне архива, без промежуточных папок.
2. **Загрузите на Яндекс.Диск:** Залейте архив на свой Яндекс.Диск и сделайте на него **публичную ссылку**.
3. **Пропишите ссылку в скрипт:** В файле `run.sh` найдите переменную `YANDEX_DISK_PUBLIC_LINK` и вставьте туда вашу ссылку:
   ```bash
   YANDEX_DISK_PUBLIC_LINK="https://disk.yandex.ru/d/ваш_код_ссылки"
   ```
4. **Загрузите изменения:** Залейте измененный `run.sh` на свой GitHub. Теперь на сервере PVE загрузчик сможет скачивать архив с Яндекс.Диска.

## Принцип работы проекта

Инфраструктура скрипта разделена на два уровня: **Загрузчик/Обновлятор (`run.sh`)** и **Интерактивное меню (`menu.sh`)**, которое вызывает специализированные проверочные скрипты для каждого варианта.

### 1. Инициализация и обновление (`run.sh`)
* Скрипт проверяет наличие root-привилегий и предлагает пользователю интерактивный выбор источника загрузки:
  1. **GitHub (по умолчанию):** использует стандартный `git pull` для обновления локальной копии в `/root/pve_checks/`. Если `git` отсутствует на хосте, скрипт автоматически переключается на поштучное скачивание через `curl` с красивым графическим прогресс-баром.
  2. **Yandex.Disk (зеркало):** обращается к публичному Yandex Cloud API, извлекает прямую ссылку, скачивает ZIP-архив репозитория во временную директорию и распаковывает его средствами встроенного в Proxmox **Python 3** (что позволяет не устанавливать дополнительные пакеты вроде `unzip`).
* После загрузки скрипт выставляет права на исполнение (`chmod +x`) и запускает меню.

### 2. Главное меню и навигация (`menu.sh`)
* Скрипт реализует двухуровневое меню:
  * **Первый уровень:** Выбор варианта экзамена (1, 2, 3 или 4) и глобальный сброс стенда (откат всех 18 ВМ).
  * **Второй уровень:** Конкретные действия по выбранному варианту (запуск проверки отдельных модулей, последовательный тест, управление питанием ВМ).
* Кнопка `0` в главном меню возвращает пользователя обратно на экран выбора варианта (благодаря бесконечному циклу `while true` в точке входа), а кнопка `0` на первом уровне завершает работу скрипта (`exit 0`).

### 3. Интеллектуальное управление питанием ВМ
* Перед началом тестирования модуля скрипт автоматически запускает выключенные ВМ (`qm start`).
* Если была запущена хотя бы одна машина, скрипт выжидает **30 секунд** с визуальным таймером обратного отсчета для загрузки операционной системы и инициализации служб (если все ВМ модуля уже были запущены, этот шаг пропускается).
* При переключении на проверку другого модуля или в режиме последовательной проверки всех модулей, ВМ ранее проверенного модуля автоматически выключаются (`qm stop`), освобождая ресурсы процессора и оперативной памяти гипервизора.
* Скрипт поддерживает гибкое групповое управление питанием (диапазоны `1-3`, `2-3`, `all` или отдельные модули). Все команды питания запускаются параллельно в фоновом режиме, а таймаут ожидания выполняется ровно один раз.

### 4. Сброс стенда
* Доступен на первом уровне меню выбора варианта.
* Скрипт последовательно обходит 18 виртуальных машин стенда (`10101-10106`, `10201-10206`, `10301-10306`).
* Проверяет существование ВМ в Proxmox VE с помощью `qm status`. Если ВМ существует, выполняет откат к исходному снапшоту: `qm rollback <vmid> exstart`.

## Принцип работы проверок (на примере Варианта 1)

Проверки в файлах `v1/module*_check.sh` выполняются в двух плоскостях: на уровне конфигурации гипервизора Proxmox VE и внутри гостевых систем.

### 1. Проверка на уровне хоста (qm config)
Некоторые параметры проверяются напрямую через парсинг файлов конфигурации виртуальных машин на самом сервере Proxmox:
* **Сетевые теги VLAN:** Для проверки правильности настройки VLAN скрипт выполняет команду `qm config <vmid>` и проверяет наличие нужного VLAN-тега (например, `tag=10`) на сетевом интерфейсе гостя (например, `net6`).
* **Аппаратная конфигурация:** Таким же образом можно валидировать настройки выделенной оперативной памяти, ядер процессора или дисковых девайсов.

### 2. Проверка внутри гостевых ОС (QEMU Guest Agent)
Основная масса тестов выполняется непосредственно внутри виртуальных машин. Для этого используется утилита интеграции **QEMU Guest Agent** (`qm guest exec`). Каждая проверка состоит из двух последовательных этапов:

1. **Запрос фактического значения (Извлечение «Факта»)**:
   * На целевой ВМ через агент запускается Bash-команда сбора данных (переменная `fact_command`).
   * *Пример для проверки IP:* `ip -br a show eth0 | awk '{print $3}'`.
   * Скрипт на хосте получает JSON-ответ от Guest Agent, парсит поля `out-data` или `err-data` и извлекает сырой текстовый вывод гостевой ОС. Это фактическое состояние выводится пользователю на экран.
2. **Верификация критерия соответствия**:
   * На целевой ВМ запускается проверочная логическая команда (переменная `check_command`).
   * *Пример для проверки IP:* `ip -br a show eth0 | grep -q '10.1.1.10/24'`.
   * Proxmox возвращает exitcode выполнения команды внутри гостя. Если `exitcode` равен `0` (команда завершилась успешно), тест считается пройденным (`[ OK ]`). В противном случае фиксируется ошибка (`[ FAIL ]`).

### 3. Логирование и формирование отчетов
* Результаты выполнения каждого теста транслируются на экран в реальном времени: пройденные тесты окрашиваются в зеленый цвет (`[ OK ]`), проваленные — в красный (`[ FAIL ]`) с указанием фактически полученного значения (например: `[ FAIL ] HQ-SRV: IP адрес eth0 (10.1.1.10/24) -> Ошибка (Факт: 10.1.1.99/24)`).
* Все строчки вывода дублируются в текстовый файл отчета в папке `/root/pve_reports/` с уникальным временным штампом в названии (например, `Module_Check_Report_21-06-2026_17-54-39.txt`). Это позволяет сохранять историю проверок для каждого студента.

## Структура проекта

Все файлы скачиваются в папку `/root/pve_checks/` и лежат там напрямую:
* `menu.sh` — главное интерактивное меню (выбор варианта экзамена и запуск модулей).
* `run.sh` — скрипт инициализации, обновления и запуска проверки по ссылке.
* `v1/`, `v2/`, `v3/`, `v4/` — папки с тестами под варианты экзамена:
  * `module1_check.sh` — проверка Модуля 1 (Сеть, базовые сервисы).
  * `module2_check.sh` — проверка Модуля 2 (Samba DC, RAID, NFS, Docker, LAMP, Nginx).
  * `module3_check.sh` — проверка Модуля 3 (ГОСТ шифрование, OpenVPN, мониторинг, бэкапы).
* `screenshots/` — скриншоты для README.

## Требования
1. Запуск от имени `root` на сервере Proxmox VE.
2. Включенный и настроенный **QEMU Guest Agent** на проверяемых виртуальных машинах (так как для проверок используется команда `qm guest exec`).
